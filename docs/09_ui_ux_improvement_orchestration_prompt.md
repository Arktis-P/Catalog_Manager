# Catalogue Manager UI/UX 개선 작업 프롬프트

이 문서는 `master` 기준 Catalogue Manager의 UI/UX를 개선하기 위한 **실행용 오케스트레이션 프롬프트**다. Codex, Claude Code 또는 동등한 코딩 에이전트에게 이 문서 전체를 전달하고 작업을 진행한다.

---

## 1. 역할과 목표

당신은 Catalogue Manager의 UI/UX 개선을 총괄하는 오케스트레이터다.

앱의 핵심 작업 흐름은 다음과 같다.

1. 캐릭터·시리즈 데이터 수집
2. 캐릭터 태그 수집·정리
3. 이미지 생성 및 재생성
4. 이미지 검수와 캐릭터 식별 확인
5. 레이팅 부여
6. 선택된 이미지를 카탈로그에 표시

최우선 목표는 **화려한 외형 변경이 아니라 수십만 개 항목을 반복 처리하는 사용자의 작업 속도, 상태 인지성, 실수 복구 가능성, 화면 밀도를 개선하는 것**이다.

Apple Design 스킬은 시각적 모방이 아니라 아래 원칙을 추출하는 참고 자료로 사용한다.

- 즉각적이고 연속적인 피드백
- 사용자가 언제든 중단·되돌릴 수 있는 상호작용
- 동작의 시작점과 종료점이 일치하는 공간적 일관성
- 상태와 계층을 명확하게 전달하는 절제된 시각 구조
- 키보드, 축소 모션, 대비를 포함한 접근성

참고 스킬:

- `https://github.com/emilkowalski/skills/blob/main/skills/apple-design/SKILL.md`

필요하면 유사한 UI/UX 스킬이나 공식 자료도 보조적으로 조사할 수 있지만, Catalogue Manager의 실제 작업 흐름과 기존 컴포넌트 구조를 우선한다.

## 2. 작업 시작 규칙

### 2.1 기준 확인

작업 시작 전 반드시 다음을 수행한다.

1. 원격 `master`를 최신 상태로 갱신한다.
2. 현재 열려 있는 PR과 UI 관련 진행 중 브랜치를 확인한다.
3. 기존 변경과 충돌하거나 중복되는 작업이 없는지 검사한다.
4. `README.md`, `USAGE_GUIDE.md`, `frontend/src/App.tsx`, `frontend/src/components/Layout.tsx`, `frontend/src/styles/global.css`, Review 관련 컴포넌트를 우선 읽는다.
5. 실제 실행이 가능하면 현재 화면을 실행해 주요 페이지와 작업 흐름을 직접 확인한다.

### 2.2 구현 브랜치

문서 브랜치가 아니라 실제 구현을 시작할 때는 최신 `master`에서 별도 브랜치를 만든다.

권장 브랜치명:

```text
feat/ui-ux-workflow-improvement
```

이미 같은 목적의 브랜치나 PR이 존재하면 새 브랜치를 만들지 말고 기존 작업을 검토한 후 이어서 사용한다.

`master`에 직접 커밋하지 않는다.

## 3. 우선 개선 범위

### P0. 정보 구조와 내비게이션

현재 기능 이름을 병렬로 나열하는 방식보다 실제 업무 흐름이 드러나도록 재구성한다.

권장 그룹:

- 작업: 수집, 생성, 리뷰
- 결과: 캐릭터 카탈로그, 시리즈 카탈로그
- 관리: 데이터 관리, 설정

세부 요구사항:

- 기존 URL과 기능 접근성을 가능한 한 유지한다.
- 메뉴 이름은 한국어 또는 프로젝트에서 정한 하나의 언어 체계로 통일한다.
- 현재 위치와 이전·다음 작업을 사용자가 즉시 파악할 수 있어야 한다.
- 자주 사용하지 않는 관리 기능은 핵심 작업과 시각적으로 분리한다.

### P0. Review 화면 재구성

Review 화면은 최우선 개선 대상이다.

목표 구조:

- 상단: 전체 진행률, 남은 항목 수, 현재 필터 요약
- 중앙: 이미지·캐릭터명·핵심 판정 중심의 검수 카드
- 고정 액션 영역: 레이팅, 저장, 재생성, 제외, 이전·다음
- 보조 영역: 프롬프트 편집, 상세 태그, 고급 필터

세부 요구사항:

- `V2`, `Catalog Review`, `Appearance`의 차이를 이름 또는 설명만 보고 이해할 수 있게 한다.
- 1차 작업에 불필요한 상세 옵션은 progressive disclosure 방식으로 접는다.
- 저장 후 다음 항목으로 자연스럽게 포커스가 이동해야 한다.
- 키보드만으로 반복 검수가 가능해야 한다.
- 현재 선택 카드와 저장되지 않은 변경 상태를 명확히 표시한다.
- 처리 중, 성공, 실패, 재생성 중 상태는 색상만으로 구분하지 않는다.
- 잘못 저장한 직후 취소할 수 있는 Undo 또는 동등한 복구 경로를 검토한다.

### P1. 전역 작업 사이드바

현재 상시 노출되는 작업 패널이 주 작업 영역을 지나치게 압박하지 않도록 개선한다.

- 접기·펼치기 지원
- 마지막 폭과 접힘 상태 저장
- 작업이 없을 때 자동 축소 또는 요약 모드
- 실행 중·실패·완료 작업 수를 접힌 상태에서도 확인
- 드래그 리사이즈 시 1:1 반응, 적절한 최소·최대 폭, 더블클릭 기본 폭 복원 검토
- 열기와 닫기의 방향·기준점을 일관되게 유지

### P1. 피드백과 상태 표현

- 버튼은 클릭 완료 이후가 아니라 누르는 즉시 시각 피드백을 준다.
- 저장·레이팅·재생성 요청은 낙관적 UI 적용 가능성을 검토하되 실패 시 원상 복구한다.
- 긴 작업은 시작 여부, 진행률, 일시정지, 실패 이유, 재시도 가능 여부를 구분한다.
- Toast만으로 중요한 상태를 전달하지 않는다.
- destructive action에는 명확한 경고 또는 즉시 Undo를 제공한다.

### P1. 디자인 시스템과 접근성

- 기존 CSS 변수와 컴포넌트를 기준으로 디자인 토큰을 정리한다.
- spacing, radius, typography, focus ring, 상태 색상의 규칙을 통일한다.
- `prefers-reduced-motion`을 지원한다.
- 키보드 포커스가 항상 보이게 한다.
- 색상 외에 아이콘, 텍스트, 형태를 함께 사용한다.
- 대비와 클릭 영역을 점검한다.
- 반투명·blur는 계층 표현에 실제 도움이 되는 영역에만 제한적으로 사용한다.

### P2. 모션

모션은 기능 개선 이후 적용한다.

- 사용자 입력 중에는 즉시 반응한다.
- 드래그·drawer·sheet 등 직접 조작 요소만 interruptible motion을 고려한다.
- 일반 버튼과 메뉴에 과도한 spring이나 bounce를 사용하지 않는다.
- transform과 opacity 중심으로 구현한다.
- 축소 모션 환경에서는 slide, bounce, parallax를 짧은 fade 또는 정적 전환으로 대체한다.

## 4. Apple Design 스킬 활용 절차

Apple Design 스킬을 그대로 복제하지 말고 다음 절차로 활용한다.

1. 스킬 문서에서 현재 변경 대상과 직접 관련된 원칙만 추출한다.
2. 각 원칙을 Catalogue Manager의 구체적인 사용자 행동과 연결한다.
3. 적용 전 아래 세 질문에 답한다.
   - 반복 검수 속도가 빨라지는가?
   - 현재 상태와 다음 행동이 더 명확해지는가?
   - 실수하거나 작업이 중단되어도 복구하기 쉬워지는가?
4. 세 질문 중 어느 것도 개선하지 못하는 시각 효과는 제외한다.
5. 적용한 원칙과 제외한 원칙을 작업 문서에 간단히 기록한다.

예시:

| 스킬 원칙 | Catalogue Manager 적용 | 적용 여부 |
|---|---|---|
| 즉각적 피드백 | 레이팅 버튼 pointer-down 피드백 | 적용 |
| interruptibility | 작업 drawer를 이동 중 다시 잡아 되돌리기 | 조건부 적용 |
| translucency | 상단 고정 도구 모음의 계층 분리 | 제한 적용 |
| momentum projection | 대량 검수 카드 이동 | 필요성 검증 후 적용 |
| 과도한 bounce | 일반 탭 전환 | 제외 |

## 5. 하위 작업자 활용 규칙

오케스트레이터가 모든 파일을 직접 읽고 구현하지 않는다. GPT-5.5, Sonnet 5 또는 사용 가능한 동급 하위 작업자를 적극 활용한다.

### 권장 분담

#### GPT-5.5 또는 강한 추론 모델

- 현재 정보 구조와 작업 플로우 분석
- Review 화면의 상태 모델·실수 경로 분석
- 단계별 구현 계획과 회귀 위험 정리
- 접근성·키보드 조작 검수
- 최종 변경의 전체 일관성 리뷰

#### Sonnet 5 또는 구현 효율이 높은 모델

- React 컴포넌트 분리와 실제 코드 구현
- CSS 토큰과 레이아웃 개선
- 테스트 작성 및 타입 오류 수정
- 반복적인 컴포넌트 정리
- 문서 업데이트 초안

#### 필요 시 추가 작업자

- 한 작업자: 기존 Review 컴포넌트와 상태 흐름만 분석
- 한 작업자: Layout·내비게이션·GlobalTaskBar만 분석
- 한 작업자: CSS·접근성·키보드·반응형만 검토
- 한 작업자: 변경 후 회귀 테스트와 사용자 시나리오 검증

### 위임 원칙

- 하위 작업자에게 저장소 전체를 무조건 읽게 하지 않는다.
- 작업에 필요한 파일 경로, 질문, 출력 형식을 명시한다.
- 동일한 파일을 여러 작업자가 중복 분석하지 않도록 범위를 나눈다.
- 구현 작업자는 자신의 변경 범위 밖의 대규모 리팩터링을 하지 않는다.
- 오케스트레이터는 하위 결과를 그대로 채택하지 말고 실제 코드와 대조한다.
- 하위 작업자의 답변은 결론, 근거 파일, 제안 변경, 위험의 네 항목으로 제한한다.

하위 작업자 요청 예시:

```text
Catalogue Manager의 Review V2 화면만 분석하라.
읽을 파일은 ReviewPage.tsx, V2ReviewPanel.tsx, V2ReviewRow.tsx 및 직접 참조되는 CSS로 제한한다.
목표는 반복 검수 속도와 실수 방지 개선이다.
출력은 1) 현재 병목 2) 최소 변경안 3) 영향 파일 4) 회귀 위험으로 작성하고 1,000자 이내로 제한하라.
코드를 수정하지 마라.
```

## 6. 토큰 절약 규칙

1. 먼저 파일 트리와 검색으로 관련 파일을 특정한 후 필요한 범위만 읽는다.
2. 이미 읽은 파일을 다시 전체 출력하지 않는다. 변경 구간과 참조 구간만 재확인한다.
3. 긴 파일은 심볼, 컴포넌트, CSS 클래스 이름으로 검색해서 필요한 줄만 읽는다.
4. 분석 결과를 장문으로 반복하지 않고 작업 체크리스트를 단일 문서에 누적한다.
5. 하위 작업자에게 전체 대화나 전체 문서를 전달하지 않는다. 필요한 배경과 파일만 제공한다.
6. 동일 문제에 여러 모델을 사용할 때는 역할을 다르게 한다. 같은 질문을 반복해서 비교하지 않는다.
7. 구현 전 완벽한 전면 재설계를 시도하지 않는다. P0 단위의 작고 검증 가능한 변경부터 적용한다.
8. 한 PR에서 불필요하게 백엔드·DB 구조를 변경하지 않는다.
9. 코드 변경 후에는 전체 파일을 설명하지 말고 변경 이유, 테스트 결과, 남은 위험만 기록한다.
10. 새 규칙을 만들기 전 기존 `README`, 사용 가이드, AI 규칙·인수인계 문서와 중복 여부를 확인한다.

## 7. 단계별 구현 방식

### Phase 0. 조사와 기준선

- 현재 주요 화면 캡처 또는 구조 기록
- 핵심 사용자 시나리오 작성
- 변경 전 빌드·테스트 상태 기록
- UI 관련 기존 이슈와 PR 확인

필수 시나리오:

1. 생성 대기 캐릭터 찾기
2. 이미지 생성 또는 재생성 요청
3. V2 Review에서 연속 10개 항목 판정
4. 잘못된 레이팅 수정
5. 실패 작업 확인 및 재시도
6. 선택된 이미지를 카탈로그에서 확인

### Phase 1. 정보 구조와 Layout

- 내비게이션 그룹화
- 메뉴 명칭 정리
- 작업 사이드바 접기·상태 저장
- 레이아웃 폭과 화면 밀도 검증

### Phase 2. Review 핵심 루프

- Review 모드 명칭·설명 개선
- 진행률·필터 요약·현재 상태 강화
- 핵심 액션 고정 배치
- 저장 후 포커스 이동과 키보드 동작 개선
- 오류·재생성·미저장 상태 표시

### Phase 3. 디자인 시스템과 접근성

- 토큰 정리
- focus-visible
- reduced motion
- 상태 표현 통일
- 클릭 영역과 대비 검수

### Phase 4. 제한적 모션과 최종 검증

- 직접 조작 요소에만 motion 적용
- 성능과 WebView2 호환성 확인
- 전체 사용자 시나리오 재검증
- 문서와 스크린샷 갱신

각 Phase는 가능하면 별도 커밋으로 남긴다. 범위가 크면 별도 PR로 분리한다.

#### `docs/ui_ux_verification.md`

- 사용자 시나리오별 검증 결과
- 키보드 조작 결과
- 접근성 확인 결과
- 빌드·테스트 결과
- 화면별 수동 확인 항목
- 사용자에게 직접 확인받아야 하는 항목

기존에 같은 목적의 문서가 있으면 새 문서를 중복 생성하지 말고 해당 문서를 확장한다.

### 세션 종료 규칙

작업 도중 중단되더라도 다음 작업자가 저장소만 읽고 이어갈 수 있어야 한다.

- 대화에만 남은 결정 사항을 문서로 옮긴다.
- 미확정 사항은 확정된 것처럼 기록하지 않는다.
- 완료되지 않은 코드는 완료로 표시하지 않는다.
- 테스트하지 않은 항목은 `미검증`으로 표시한다.
- 마지막 커밋과 다음 작업을 한 줄로 명확히 남긴다.

## 9. 구현 제약

- 기존 API 계약과 데이터 모델을 변경하지 않는 것을 기본으로 한다.
- React 19, React Router, 현재 TypeScript·Vite 구조를 유지한다.
- 새 UI 라이브러리나 모션 라이브러리는 실제 이점과 번들 비용을 검토한 뒤 최소한으로 추가한다.
- 단순 탭·버튼 효과를 위해 대형 라이브러리를 추가하지 않는다.
- WebView2 데스크톱 앱에서 동작해야 한다.
- 기존 단축키와 저장 동작을 깨뜨리지 않는다.
- 사용자 설정은 가능한 한 기존 settings API 또는 local storage 정책과 일관되게 저장한다.
- 대량 목록에서 불필요한 전체 재렌더링과 고비용 blur를 피한다.

## 10. 완료 기준

다음 조건을 충족해야 UI/UX 개선 작업이 완료된 것으로 본다.

- 주요 메뉴가 실제 업무 흐름 기준으로 이해된다.
- Review 화면에서 사용자가 설명 문서를 다시 읽지 않고도 다음 행동을 판단할 수 있다.
- 키보드만으로 연속 검수가 가능하다.
- 저장·재생성·실패·미저장·완료 상태가 명확하다.
- 작업 사이드바가 주 작업 공간을 불필요하게 침범하지 않는다.
- 잘못된 행동을 즉시 복구할 방법이 있다.
- 축소 모션과 키보드 포커스를 지원한다.
- 기존 핵심 기능과 API 동작에 회귀가 없다.
- 빌드와 기존 테스트가 통과한다.
- 계획·인수인계·검증 문서가 현재 코드와 일치한다.

## 11. 최종 보고 형식

최종 보고는 다음 형식으로 간단히 작성한다.

```markdown
# UI/UX 개선 결과

## 완료
- ...

## 주요 변경
- 사용자 흐름:
- Review:
- 작업 패널:
- 접근성:

## 검증
- Build:
- Test:
- Manual:

## 사용자 확인 필요
- ...

## 남은 작업
- ...
```

PR은 처음에는 Draft로 생성하고, 각 완료 기준과 검증 문서가 충족된 뒤 Ready for review로 전환한다.
